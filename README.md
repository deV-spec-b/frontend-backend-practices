##  О проекте

Это учебный бэкенд-сервис, который начинался с простого API на Node.js, а в итоге превратился в полноценное микросервисное приложение в контейнерах. В процессе разработки база данных сменилась с PostgreSQL на MongoDB, добавилось кэширование через Redis, появилась балансировка нагрузки через Nginx и HAProxy, а затем всё приложение упаковали в Docker и научились запускать одной командой в WSL.

## Практики 19–20: Базы данных (PostgreSQL → MongoDB)

### PostgreSQL (практика 19)
Сначала я подключил к серверу реляционную базу PostgreSQL через библиотеку `pg`.  
Создал таблицу пользователей с полями: `id`, `first_name`, `last_name`, `age`, `created_at`, `updated_at`.  
Написал полный набор CRUD-эндпоинтов:
- `POST /users` — создание
- `GET /users` — список всех
- `GET /users/:id` — получение одного
- `PATCH /users/:id` — обновление
- `DELETE /users/:id` — удаление

### MongoDB (практика 20)
Переписал тот же API для MongoDB с использованием Mongoose.  
Схема осталась похожей, но база стала **документо-ориентированной** и гибкой.  
Вместо SQL-запросов использовал методы Mongoose: `save()`, `find()`, `findById()`, `findByIdAndUpdate()`, `findByIdAndDelete()`.

## Практика 21: Кэширование с Redis

Добавил Redis, чтобы ускорить частые запросы.

### Принцип работы:
1. GET-запрос сначала проверяет наличие данных в Redis.
2. Если данные есть — возвращаются **мгновенно из оперативной памяти** (`source: "cache"`).
3. Если данных нет — сервер идёт в базу, отдаёт ответ и **сохраняет его в Redis** на заданное время.

### Время жизни кэша (TTL):
- Список пользователей — **1 минута**
- Список товаров — **10 минут**

### Очистка кэша
При изменении или удалении данных (`POST`, `PUT`, `DELETE`) кэш **автоматически очищается**, чтобы клиенты не видели устаревшую информацию.

## Практика 22: Балансировка нагрузки

Я запустил несколько одинаковых backend-серверов (3 экземпляра) и настроил балансировщики.

### Nginx
- Алгоритм: **Round Robin** (запросы распределяются по кругу)
- Параметры отказоустойчивости:  
  `max_fails=2 fail_timeout=30s` — после 2 ошибок сервер исключается на 30 секунд
- Резервный сервер: `server backend3:3000 backup`

### HAProxy (альтернативный балансировщик)
- Алгоритм: **roundrobin**
- Проверка доступности: параметр `check`
- Порт: `8080`

### Что проверил:
-  Запросы чередуются между серверами
-  При остановке одного контейнера — трафик идёт на работающие
-  При отказе всех основных — активируется резервный сервер (`backup`)

##  Практика 23: Контейнеризация с Docker

Всё приложение упаковал в контейнеры.

### Что сделано:
1. Написал **Dockerfile** для каждого сервиса на основе лёгкого образа `node:18-alpine`.
2. Создал **`docker-compose.yml`**, где описал:
   - 3 backend-сервера
   - Nginx (балансировщик)
   - HAProxy (альтернативный балансировщик)
   - (опционально) Redis, PostgreSQL, MongoDB
3. Все сервисы объединены в **общую сеть** и общаются друг с другом **по именам контейнеров** (например, `backend1:3000`).
4. Данные баз сохраняются между перезапусками благодаря **томам (volumes)**.

### Команды для управления:
bash
# Запуск всех сервисов
docker compose up -d --build

# Остановка
docker compose down

# Просмотр логов
docker compose logs -f
